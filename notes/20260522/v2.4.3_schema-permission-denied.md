# Note 2026-05-22 v2.4.3 — 修 v2.4.2 Step 6b Permission denied

## 情境

USER 跑 v2.4.2_install-portal-all-in-one.sh, Step 6b 直接 FAIL:

```
=== Step 6b: 套用 Schema + 驗 6 個關鍵表必須在 ===
[info] 套用 schema: /tmp/sf-portal-extracted/portal/sql/01_create_db_postgres.sql
psql: error: /tmp/sf-portal-extracted/portal/sql/01_create_db_postgres.sql: Permission denied
[warn] 缺 table: auditlog
[warn] 缺 table: businesscode
...
[FAIL] 缺 6 個表
```

比 v2.4.1 慘 — v2.4.1 至少 portal 起來 (有 502 但 systemctl active), v2.4.2 因為 Step 6b fail 整個中斷, portal 根本沒裝完.

## Root cause

`sudo -u postgres psql -d $DB_NAME -f $SCHEMA_SQL`

- root tar xzf 解壓到 `/tmp/sf-portal-extracted/`, owner = root, mode 預設 700/750
- postgres user 沒權限讀這個目錄 → `Permission denied`
- psql 沒讀到 SQL 檔 → 表沒建 → MISSING_TBL=6 → fail

`/tmp` 通常是 1777 (sticky bit + world read), 但 root 解壓出來的子目錄 owner 是 root, 預設 mode 跟 umask 走 (通常 022 → 755 應該 OK, 但有時 0700 看 tar 預設).

實際情況: `/tmp/sf-portal-extracted/` 可能 700, 或裡面 portal/sql/ 是 700, postgres user 進不去.

## v2.4.3 修法

### Step 6b 改 3 段 fallback

**第 1 段**: 先 cp 到 `/tmp/sf-schema-$$.sql`, chown postgres, mode 644 → 大部分情況 OK

```bash
SCHEMA_TMP="/tmp/sf-schema-$$.sql"
cp "$SCHEMA_SQL" "$SCHEMA_TMP"
chmod 644 "$SCHEMA_TMP"
chown postgres:postgres "$SCHEMA_TMP" 2>/dev/null || true
sudo -u postgres psql -d "$DB_NAME" -f "$SCHEMA_TMP" 2>&1 | tail -15
rm -f "$SCHEMA_TMP"
```

**第 2 段**: 驗 6 個表都在 (不變)

**第 3 段**: 若仍缺, cp 到 `/var/lib/pgsql/` (postgres home), 一定有權限, 再 retry 一次

```bash
PG_HOME=$(getent passwd postgres | cut -d: -f6)
RETRY_SQL="$PG_HOME/sf-schema-retry.sql"
cp "$SCHEMA_SQL" "$RETRY_SQL"
chown postgres:postgres "$RETRY_SQL"
sudo -u postgres psql -d "$DB_NAME" -f "$RETRY_SQL" 2>&1 | tail -30
```

### 為什麼不用 cat | psql

我原本想用 `cat $SCHEMA_SQL | sudo -u postgres psql ...` (root 讀檔, pipe 給 postgres stdin).

但這個有風險:
- `\copy` 之類的 meta-command 在 stdin pipe 模式可能行為不同 (我們 schema 沒用, 但保守)
- 錯誤訊息行號定位不準 (沒 -f 不知道是哪行炸)

所以還是用 `-f` + 拷檔到 postgres 可讀位置.
