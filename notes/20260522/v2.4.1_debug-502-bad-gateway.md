# Note 2026-05-22 v2.4.1 — Debug 502 Bad Gateway

## 情境

USER 跑完 v2.4.1 安裝, 瀏覽器開 `http://10.92.198.14/auth/login?next=%2F` 顯示:

```
502 Bad Gateway
nginx/1.20.1
```

`/auth/login` 這個 URL 代表 **Portal 之前有跑成功過**, 現在 gunicorn 掛了或斷線.

## 502 意義 (記在腦裡)

502 = nginx 接到請求但**連不到 upstream** (我們的 127.0.0.1:5000 gunicorn)

3 種可能:
- A: gunicorn 掛了 / 沒起來
- B: gunicorn 起來但 bind 不在 5000 / port 衝突
- C: gunicorn 起來且 bind 5000, 但 nginx → 5000 中間被擋 (SELinux 最常見)

## 3 條診斷指令 (USER 手打)

### 1. 看 sf-portal 狀態

```
systemctl status sf-portal --no-pager
```

關注:
- `Active: active (running)` ✅
- `Active: failed` ❌ → 看 error.log
- `Active: activating (auto-restart)` ❌ → worker boot loop

### 2. 看 gunicorn 是否真的 listen 5000

```
ss -tlnp | grep 5000
```

關注:
- 有看到 `127.0.0.1:5000 ... gunicorn` ✅
- 沒輸出 ❌ → gunicorn 沒 listen
- 看到別的 process 佔 5000 ❌

### 3. 看 gunicorn 內部錯誤

```
sudo tail -30 /opt/portal/logs/error.log
```

找關鍵字 `Traceback` 或 `[ERROR]`.

## 對照表

| `systemctl status` | `ss 5000` | `error.log` | 診斷 | 修法 |
|---|---|---|---|---|
| active | 有 | 乾淨 | C: SELinux 擋 nginx | 跳到「修 C」 |
| active | 無 | 有 OSError | port 衝突 | 跳到「修 B」 |
| failed | 無 | Python Traceback | gunicorn 又掛 | 把 traceback 截圖貼 GPT/Claude |
| activating | 無 | 連續 worker fail | code 有問題 | 看 traceback |

## 修 A: gunicorn 掛了

把 `error.log` 內第一個 Traceback 截圖貼上來,或跑這條撈完整 traceback:

```
cd /opt/portal/app && sudo -u nginx /usr/bin/gunicorn --workers 1 --bind 127.0.0.1:5001 --log-level debug --chdir /opt/portal/app wsgi:app
```

按 Ctrl+C 中止. stdout 印的 Traceback 對症修.

## 修 B: gunicorn bind 失敗 (port 5000 被佔)

```
sudo ss -tlnp | grep 5000
```

看誰佔的 (PID). 若不是 gunicorn:

```
sudo kill <PID>
sudo systemctl restart sf-portal
```

## 修 C: SELinux 擋 (最常見)

### 確認是 SELinux

```
getenforce
```

回 `Enforcing` 才有可能.

```
sudo ausearch -m AVC --start recent
```

或

```
sudo journalctl -t setroubleshoot --since '10 min ago' --no-pager | head -20
```

關鍵字找 `denied` + `http_port_t` / `httpd_can_network_connect`.

### 修

```
sudo setsebool -P httpd_can_network_connect on
sudo systemctl restart nginx
```

也可改抓:

```
sudo ausearch -m AVC --start recent | head
```

若仍擋, 看是不是 `name_connect` 之類, 可暫時 permissive 測:

```
sudo setenforce 0
curl -I http://localhost/
sudo setenforce 1
```

若 permissive 下 OK → 確定是 SELinux. 用 audit2allow 客製 policy:

```
sudo ausearch -m AVC --start recent | audit2allow -M sf-portal-nginx
sudo semodule -i sf-portal-nginx.pp
```

## 補丁: nginx 本身 (萬一)

### 確認 nginx config

```
sudo nginx -t
```

### 看 nginx 端錯

```
sudo tail -30 /var/log/nginx/sf-portal-error.log
```

常見錯:
- `connect() to 127.0.0.1:5000 failed (13: Permission denied)` → SELinux (跳「修 C」)
- `connect() to 127.0.0.1:5000 failed (111: Connection refused)` → gunicorn 沒在 5000 (跳「修 A」或「修 B」)
- `upstream prematurely closed connection` → gunicorn 起來但 worker 處理 request 中崩, 看 error.log

### 強制重啟兩邊

```
sudo systemctl restart sf-portal && sleep 3 && sudo systemctl restart nginx && curl -I http://localhost/
```
