# Note 2026-05-22 v2.3.9 — 找 systemd 跑 gunicorn 的真正 Python error

## 情境

USER 跑 v2.3.8:
- Step 1-8 全 ✅
- Step 9 sf-portal status=1 (Worker failed to boot)
- 你手動跑 `sudo -u nginx /usr/bin/gunicorn ... wsgi:app` 看到 `No module named 'wsgi'`
  → **但這是因為你在 /tmp 跑 (gunicorn config 顯示 `chdir: /tmp`), 不是真正錯誤**
  → systemd 跑有 `WorkingDirectory=/opt/portal/app`, 不會有這個

systemd 真正錯誤被 gunicorn 的 HaltServer wrap 蓋掉了, 必須看別處。

## 2 個指令找真正錯誤 (依序跑)

### 1. 看 gunicorn 自己的 error log (完整 Python traceback)

```
cat /opt/portal/logs/error.log
```

這個檔是 gunicorn 的 `--error-logfile`, 內含 worker boot 失敗時的**完整 Python traceback**。

要找的關鍵字: 第一個 `Traceback (most recent call last):` 那段.

### 2. 在正確目錄手動跑 gunicorn (chdir 對才行)

```
cd /opt/portal/app
sudo -u nginx /usr/bin/gunicorn --workers 1 --bind 127.0.0.1:5001 --log-level debug --chdir /opt/portal/app wsgi:app
```

注意:
- 先 `cd /opt/portal/app` (不要在 /tmp 跑)
- 加 `--chdir /opt/portal/app` 雙保險
- 按 Ctrl+C 中止

## 預期會看到 (從之前 db.py 重寫經驗推測)

| 可能錯誤 | 原因 | 修法 |
|---|---|---|
| `ModuleNotFoundError: No module named 'XXX'` | 又缺套件 | 補對應 PyPI wheel |
| `psycopg2.OperationalError: FATAL: password authentication failed` | DB 密碼不對 | 重設 portal user password |
| `psycopg2.OperationalError: FATAL: database "file_exchange_audit" does not exist` | DB 沒建 | 看 Step 6 為何失敗 |
| `psycopg2.errors.UndefinedTable: relation "businesscode" does not exist` | schema 沒套用, 表沒建 | 跑 sql/01_create_db_postgres.sql |
| `KeyError: 'DB_CONNECTION_STRING'` | config.py 讀 appsettings.json 寫法不對 | 看 config.py |
| `FileNotFoundError: '.env'` | dotenv 找不到 .env 檔 | 確認 .env 路徑 |
