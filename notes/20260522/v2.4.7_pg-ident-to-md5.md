## Root Cause (error.log)

```
psycopg2.OperationalError: FATAL:  Ident authentication failed for user "portal"
```

Portal 程式以 `nginx` user 跑, 連 `127.0.0.1:5432` 走 PG 的 `ident` 認證 → PG 去查 OS user 是否叫 `portal` → 不是 → reject.

我 v2.4.6 安裝腳本有加 `host file_exchange_audit portal 127.0.0.1/32 md5` 到 pg_hba.conf, **但加在尾巴**. PG 是 first-match-wins, 預設第一條 `host all all 127.0.0.1/32 ident` 先 match 就用 ident, 永遠走不到我那條.

## 為什麼有些 200 有些 500?

v2.4.5 我加了 try/except wrap 大部分 route — DB 連不到就 catch → flash + render 空 list → HTTP 200 但**沒資料**.

`biz_edit` 我**沒**加 try/except → DB error 噴出 → 500.
`/api/health` 故意設計 DB fail 回 503 (給 monitoring 用).

## v2.4.7 修法

### 1. pg_hba.conf: 所有 ident/peer → md5

用 Python 解析 pg_hba.conf, 把 host/local 行最後欄位是 `ident` / `peer` 的全改成 `md5`:

```
原 (預設):
local   all             all                                     peer
host    all             all             127.0.0.1/32            ident
host    all             all             ::1/128                 ident

改成:
local   all             all                                     md5
host    all             all             127.0.0.1/32            md5
host    all             all             ::1/128                 md5
```

另外確保 portal user 專屬規則在最頂.

### 2. biz_edit 加 try/except
USER 點「編輯」按鈕不再 500.

### 3. systemctl reload postgresql + restart sf-portal

腳本內建測 `PGPASSWORD=... psql -U portal -d file_exchange_audit -h 127.0.0.1 -c "SELECT 'OK'"` 驗證認證通了才往下.

### 4. Step 9 重掃 route 印表格

USER 不用一個個試, 一眼看哪個 route 還掛.

## 反省

我 v2.4.0 寫 install 腳本時加 pg_hba 規則 — 用 `>>` 追加到尾巴, 沒去動預設那 3 行 (local/host/host all all ident).

`first-match-wins` 是 PG `pg_hba.conf` 的基本特性, 我假設「加到尾巴自然會 match」是錯的.

下次寫 pg_hba 變更要:
1. 看現有規則 (`SELECT * FROM pg_hba_file_rules` 在 PG 內也能查)
2. 規則加到要 match 的 user/db 的 ident/peer 規則**之前**
3. 或直接改預設規則 (這次我就這樣解)

SKILL 鐵律 12.5 (擬):
**改 pg_hba.conf 要看現有規則順序, 不要無腦 append.**
