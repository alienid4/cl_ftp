# Note 2026-05-21 v2.2.3 — Portal 沒起來時 3 個診斷指令

## 情境

USER 跑完 `fix-portal-v2.2.3.sh` 印 DONE, 但:
- `ss -tlnp | grep 5000` 沒輸出 (Port 5000 沒監聽)
- 代表 sf-portal.service 沒真的起來

## 3 個診斷指令 (依序在 SF 主機跑)

### 1. systemctl 看 service 狀態

```
systemctl status sf-portal -l --no-pager
```

> 注意: RHEL 9 是 `systemctl`, 不能用 `service status` (老舊 SysV 指令, RHEL 9 只支援 start/stop/restart)

預期看到:
- `Active: active (running)` ✅ → port 5000 應該有, 看下個指令
- `Active: failed` ❌ → 看下方 ExecStart 訊息 + journalctl
- `Loaded: not-found` ❌ → service unit 沒寫成功, fix_portal Step 4 失敗

### 2. journalctl 看最近 50 行 log

```
journalctl -u sf-portal -n 50 --no-pager
```

預期看到 gunicorn 的 startup log 或 Python traceback.

最常見錯誤:
| 訊息 | 原因 | 修法 |
|---|---|---|
| `ModuleNotFoundError: No module named 'flask'` | EPEL 沒裝起來 | 重跑 fix_portal Step 1b |
| `psycopg2.OperationalError: FATAL: password authentication failed` | appsettings.json 密碼錯 | 重寫 appsettings.json |
| `ImportError: cannot import name 'app' from 'wsgi'` | Portal source 拷錯 / wsgi.py 內容不對 | 看 /opt/portal/app/wsgi.py |
| `Address already in use` | 5000 port 被佔 | `lsof -i :5000` 看誰用 |
| `permission denied: /opt/portal/logs/...` | log 目錄 owner 不對 | `chown -R nginx:nginx /opt/portal` |

### 3. tail stderr log

```
tail -50 /opt/portal/logs/portal-stderr.log
```

跟 journalctl 互補, 看 gunicorn 自己寫的錯。
