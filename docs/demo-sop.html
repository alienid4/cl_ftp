⚙️ 角色 2️⃣ — SF 排程 (背景自動跑, 不用操作) batch_aggregator systemd timer 每 5 秒掃
3
等 30 秒 — 排程聚合批次

解說邏輯給主管:

  • 每 5 秒 systemd timer 掃 /data/exchange/u0X/inbound/ 新檔
  • 同 u0X + 同來源 IP + 30 秒內視為「同一批」
  • 30 秒沒新檔 → batch 自動關閉 → status: OPEN → PENDING_APPROVAL
  • 5 分鐘安全閥: 同一批最多 5 分鐘, 超過強制關閉 (避免持續上傳永遠不結案)
等待約 35 秒 讓 idle close 觸發。期間可講解 ANY 制簽核設計。

⚠️ 確認: 全收 3 個檔

點下「🚀 確認執行」後, 系統會:

  1. UPDATE batch SET status=APPROVED, approved_by=你
  2. 把 3 個檔從 /data/exchange/u01/inbound/ 搬到 \\SF\architecture\
  3. 寫 AuditLog: APPROVE_OK × 1 + MOVE_OK × 3
  4. 寄信通知 dept_archi_view 群組可下載
  5. redirect 回待簽列表, 該批消失

⚠ 檔搬走後要復原需 IT 從 samba 手動搬回. AuditLog 永遠留紀錄.
⚠ 其他 4 位簽核人之後看到「已被你處理」, 不能再簽.

勾 checkbox → 「🚀 確認執行」按鈕從灰 (disabled) 變紅 (enabled)。

🎯 收尾 + 結論

✅ 本次 demo 證明

  • SFTP chroot 隔離 — u01 看不到其他帳號目錄
  • 排程聚合 + 安全閥 — 滑動 30s 同批,5 分鐘強制 close
  • 真實 AD bind — LDAP simple bind 兩段式 (服務帳號查 DN → user DN+pwd 驗證)
  • 失敗訊息明確 — 不讓 USER 猜「帳密錯哪個」
  • ANY 制 5 人簽核 — 任 1 同意即放行
  • 二步預覽防誤觸 — 副作用攤開 + 勾 checkbox 才能執行
  • 自動寄通知信 — 簽核完瞬間 OA USER 收到
  • SMB 下載 OA 友善 — 用熟悉的 Windows 檔案總管
  • 完整稽核 — 全公司操作留痕,匯出 CSV / 接 SIEM

📋 上線前還要做 (給主管看路線圖)

項動作大概時間
1. 真實 AD 接.env 改 AD_SERVER / AD_BIND_USER, 開 AD 服務帳號工單1 天 (含等 AD 管理員)

8. 備份排程Windows Server Backup or rsync + 設定檔 + ACL dump1 天
